# CoinGecko API Test Suite

![fast tests](https://github.com/YOUR_USERNAME/YOUR_REPO/actions/workflows/fast.yml/badge.svg)

---

Проект автотестов для публичного [CoinGecko API v3](https://www.coingecko.com/en/api/documentation), написанный на Python + pytest. Покрывает три эндпоинта: `/ping`, `/coins/markets`, `/coins/{id}` — всего 25 тест-кейсов.

Цель проекта — показать подход к организации API-тестов, который масштабируется и удобен в поддержке: понятная структура, CI/CD из коробки, никаких хардкодов.

## Что применяется и почему

**Разделение на fast/slow тесты**
Тесты помечены маркерами `@pytest.mark.fast` и `@pytest.mark.slow`. Fast — один запрос, быстрый ответ — гоняются на каждый коммит. Slow — параметризованные и многошаговые — раз в день по расписанию. Это стандартная практика в командах: не заставлять разработчика ждать минуту после каждого пуша.

**Параметризация через `@pytest.mark.parametrize`**
Там где одна и та же логика проверяется для нескольких входных данных (разные валюты, разные монеты), используется параметризация. Каждый набор параметров — отдельный тест-кейс в отчёте. Если сломается EUR — USD продолжит выполняться и будет видно что именно упало.

**Фикстуры с правильным scope**
`api_client` — `scope="session"`, одна HTTP-сессия на весь прогон. TCP-соединение переиспользуется, а не открывается заново для каждого теста. При запуске с `-n auto` (xdist) каждый воркер получает свою копию — именно так и должно работать.

**Параллельное выполнение через pytest-xdist**
`pytest -n auto` распределяет тесты по всем доступным ядрам. В CI это заметно сокращает время прогона. Работает вместе с retry-механизмом, который обрабатывает 429 от rate limit CoinGecko.

**Retry для нестабильных сетевых запросов**
`pytest-rerunfailures` повторяет упавший тест до 3 раз с паузой 2 секунды. Публичные API иногда отвечают медленно или возвращают 429 — без retry такие тесты были бы ложными фейлами в CI.

**Логирование запросов при падении**
Autouse-фикстура перехватывает упавшие тесты и печатает полный дамп запроса и ответа: метод, URL, заголовки, тело, статус, время. В CI это экономит время — не нужно воспроизводить падение локально чтобы понять что пошло не так.

**Поддержка нескольких окружений**
Переменная `ENV` указывает какой `.env`-файл загружать. `ENV=dev pytest` — локальная разработка, `ENV=prod pytest` — прод или CI. Секреты не коммитятся, в репозитории только `.env.example` как шаблон. В CI значения приходят из GitHub Actions secrets.

**HTTP-клиент вынесен в отдельный класс**
`CoinGeckoClient` — тонкая обёртка над `requests.Session`. Base URL, таймаут, заголовки и API ключ задаются один раз. Тесты описывают что проверить, не как сделать запрос.

## Структура проекта

```
coingecko-api-tests/
├── .github/
│   └── workflows/
│       ├── fast.yml         # запускается на каждый пуш и PR
│       └── full.yml         # полный прогон раз в день или вручную
├── tests/
│   ├── conftest.py          # фикстуры, логирование при падении
│   ├── test_ping.py         # GET /ping
│   ├── test_markets.py      # GET /coins/markets
│   └── test_coin.py         # GET /coins/{id}
├── utils/
│   ├── api_client.py        # HTTP-клиент
│   └── config.py            # загрузка .env.{ENV}
├── .env.example             # шаблон — скопировать в .env.dev / .env.prod
├── .gitignore
├── pytest.ini
└── requirements.txt
```

## Быстрый старт

```bash
python -m venv .venv
source .venv/bin/activate      # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env.dev
```

---

## Запуск тестов

```bash
pytest -m fast              # только быстрые
pytest -m fast -n auto      # быстрые, параллельно
pytest -n auto              # всё, параллельно
ENV=prod pytest -n auto     # против прод-окружения
```

## CI/CD

| Воркфлоу | Триггер | Что запускает |
|----------|---------|---------------|
| `fast.yml` | Каждый пуш и PR (Python 3.11 и 3.12) | `pytest -m fast -n auto` |
| `full.yml` | Каждый день в 06:00 UTC или вручную | `pytest -n auto` |
